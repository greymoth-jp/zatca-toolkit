Documentation

Generate, validate, sign and (gated) clear ZATCA Fatoora Phase 2 invoices — in Node or the browser, in-process. The invoice never leaves you.

VERIFIED IN CIEvery code snippet on this page is the exact code in packages/sdk/examples/api-tour.mjs, executed on every push by test/api-tour.test.mjs. If a snippet drifts from the engine, CI goes red — these examples cannot silently rot.

Install

Node 20+ or any modern browser. The deterministic engine ships as WebAssembly inside the package — no native build, no server.

shell
npm install @zatca/sdk

Quickstart

Generate ZATCA-UBL from a normalized invoice, then check it would clear — all in-process.

quickstart.mjs
import { Zatca } from "@zatca/sdk";

const z = new Zatca({ mode: "sandbox" });
const { ubl } = await z.generate(sampleInvoice);   // normalized -> UBL 2.1 XML
const r = await z.validateXML(ubl, "zatca-ksa");   // deterministic ZATCA / EN16931
console.log(r.report.valid ? "would clear" : r.report.errors);
// -> would clear        (sample invoice: valid, 0 findings)

The normalized invoice model

One plain object describes the invoice; the engine renders the UBL and runs the rules. A KSA standard tax invoice (type_code 388; use 381 for a credit note):

sampleInvoice
const sampleInvoice = {
  profile_id: "reporting:1.0",
  id: "INV-2026-00042",
  issue_date: "2026-06-14", issue_time: "10:30:00",   // ZATCA requires the time too
  type_code: "388",                 // 388 tax invoice · 381 credit note
  currency: "SAR", tax_currency: "SAR",
  seller: { name: "Acme Trading LLC", name_ar: "شركة أكمي للتجارة",
            vat_id: "300000000000003", country_code: "SA" },
  buyer:  { name: "Beta Retail Co", vat_id: "311111111111113", country_code: "SA" },
  lines: [{ id: "1", quantity: 2, unit_code: "PCE", item_name: "Widget",
            net_price: 50, net_amount: 100, vat_category: "S", vat_rate: 15 }],
  tax_breakdown: [{ category: "S", rate: 15, taxable_amount: 100, tax_amount: 15 }],
  totals: { line_extension_amount: 100, tax_exclusive_amount: 100,
            tax_amount: 15, tax_inclusive_amount: 115, payable_amount: 115 },
};

generate(doc)

Render the normalized model to ZATCA-UBL 2.1 XML.

generate
const { ok, ubl } = await z.generate(sampleInvoice);
// ok:  true
// ubl: "<?xml version=\"1.0\" ...><Invoice ...>...</Invoice>"

validateXML(xml, profile) & validate(doc, profile)

Run the deterministic rule set against either an XML string or the normalized object. The return is a report with a boolean and an array of findings.

validate
const r = await z.validateXML(ubl, "zatca-ksa");
// { ok: true, report: { valid: true, errors: [] } }

// or validate the object directly, no XML in hand yet:
const r2 = await z.validate(sampleInvoice, "en16931");
// { ok: true, report: { valid: boolean, errors: [...] } }
profilewhen to use
"zatca-ksa"ZATCA Fatoora Phase 2 (KSA) — EN16931 + ZATCA-KSA rules.
"en16931"EN16931 semantic core only (cross-border baseline).
"peppol-bis"EN16931 + Peppol BIS rules (default).

Reading findings

Every finding is actionable: a stable rule id, the offending path, a bilingual message, and a severity. No cryptic codes.

findings
for (const e of r.report.errors) {
  console.log(e.rule_id, e.severity, e.message_en);
  // e.g.  BR-CO-15  fatal  "Invoice total with VAT must equal ..."
}
fieldmeaning
rule_idStable id, e.g. BR-CO-15, BR-KSA-ST-PIH.
pathLocation of the problem in the document.
message_en / message_arHuman message in English and Arabic.
severity"fatal" or "warning".
fix_en / fix_arHow to fix it (present on structural & QR findings).

validateStructure(xml)

Check the ZATCA structural elements of a submitted invoice — UUID, ICV, PIH, QR and the signature. Plain UBL has none of these yet, so this is what you see before signing:

structure
import { loadEngine } from "@zatca/sdk";
const engine = await loadEngine();

const s = engine.validateStructure(submittedXml);
// { ok: false, findings: [
//   { rule_id: "BR-KSA-ST-UUID" }, { rule_id: "BR-KSA-ST-ICV" },
//   { rule_id: "BR-KSA-ST-PIH" },  { rule_id: "BR-KSA-ST-QR" },
//   { rule_id: "BR-KSA-ST-SIG" } ] }

validateQR(qr, opts)

Verify a ZATCA QR (Base64 TLV). A malformed QR returns a finding carrying an English and Arabic fix hint.

qr
const q = engine.validateQR(qrBase64, { signed: true, simplified: false });
// { ok: false, findings: [{
//     rule_id: "BR-KSA-QR-01",
//     message_en: "...", message_ar: "...",
//     fix_en: "...", fix_ar: "..." }] }

clear(signedXml, opts) — credential-gated

CREDENTIAL-GATEDThis SDK does not operate hosted clearance. Live clearance requires your onboarded EGS + ZATCA Production CSID, or a certified partner. The default clearer is a clearly-labeled mock that never claims a real clearance.

Clearance is idempotent on idempotencyKey: a repeated key replays the first result and can never double-clear. Inject your own clearer for production.

clear
const z = new Zatca({ mode: "sandbox" /*, clearer: myCertifiedPartnerClearer */ });

const first  = await z.clear(signedXml, { idempotencyKey: "INV-2026-00042" });
// { status: "cleared", uuid: "INV-2026-00042", mock: true, note: "MOCK clearance ..." }

const replay = await z.clear(signedXml, { idempotencyKey: "INV-2026-00042" });
// { ..., idempotentReplay: true }   // same key -> never a second clearance

Webhooks

HMAC-SHA256 sign/verify for your own clearance callbacks. Verify against the raw bytes — do not re-serialize the JSON.

webhooks
import { signWebhook, verifyWebhook } from "@zatca/sdk";

const sig = signWebhook(rawBody, process.env.WHSEC);   // "hmac-sha256=..."
const ok  = verifyWebhook(rawBody, req.headers["x-signature"], process.env.WHSEC);
// ok: true   (timing-safe; false on tampered body or wrong secret)

Browser / WASM

The same WebAssembly engine runs in the browser — this is exactly how the free auditor validates without a server. A pass in the browser is the same pass the SDK and CI see. Bundle @zatca/sdk/wasm/engine.wasm as an asset and call loadEngine().
